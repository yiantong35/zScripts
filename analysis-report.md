## 三、可行性评估

### 3.1 P0优先级可行性分析

#### 1. 执行器非阻塞模型修复 ✅ 可行

**当前状态**：
- executor.zig 使用 ring buffer 存储输出
- poll() 直接调用 child.wait()（阻塞）
- 只有激活标签页轮询

**需改动文件**：
- `src/core/executor.zig` - 重构poll()逻辑
- `src/gui/app.zig` - 改进轮询机制
- `src/main.zig` - 可能需要事件循环调整

**技术难度**：中等
- 需要使用 std.posix.poll() 或 select()
- 需要处理非阻塞IO
- 需要管理进程组

**风险**：低（改进现有逻辑，不破坏API）

#### 2. JSON序列化和写盘策略 ✅ 可行

**当前状态**：
- config.zig 有 CONFIG_WRITE_DEBOUNCE_MS (300ms)
- 但缺少原子写入保证
- 直接写入目标文件（无临时文件）

**需改动文件**：
- `src/storage/config.zig` - 添加原子写入

**技术难度**：低
- 标准做法：写临时文件 → fsync → rename
- Zig std库支持

**风险**：低（改进数据安全，向后兼容）

#### 3. app.zig拆分重构 ✅ 可行

**当前状态**：
- app.zig 2015行，占代码库52%
- 混合了UI、状态管理、事件处理

**需改动文件**：
- `src/gui/app.zig` - 主状态管理（~300行）
- 新建 `src/gui/components/` - 拆分UI组件
- 新建 `src/gui/utils/` - 辅助函数

**技术难度**：中等
- 需要理解当前代码逻辑
- 需要定义清晰的组件边界
- 需要保持状态管理一致性

**风险**：中等（大规模重构，需充分测试）

#### 4. 添加单元测试 ✅ 可行

**当前状态**：
- build.zig 有测试基础设施
- 但没有任何 *_test.zig 文件
- 项目已有 std.testing 导入

**需改动文件**：
- 新建 `src/core/script_test.zig`
- 新建 `src/core/executor_test.zig`
- 新建 `src/core/scanner_test.zig`
- 新建 `src/storage/config_test.zig`

**技术难度**：低
- Zig std.testing 相对简单
- 可从简单模块开始

**风险**：低（新增文件，不影响现有代码）

### 3.2 P1优先级可行性分析

#### 搜索/过滤功能 ✅ 可行

**需改动文件**：
- `src/gui/app.zig` - 添加搜索UI和过滤逻辑
- `src/core/scanner.zig` - 可能需要索引优化

**技术难度**：低-中等
- 简单字符串匹配即可满足初期需求
- 可后续优化为模糊搜索

**风险**：低

#### 执行历史记录 ✅ 可行

**需改动文件**：
- `src/storage/config.zig` - 添加历史记录持久化
- `src/gui/app.zig` - 添加历史UI

## 五、优先级排序与理由

### P0（关键，必须立即处理）

1. **执行器非阻塞模型修复** ⭐⭐⭐
   - 理由：影响UI响应性，是"非阻塞执行"的核心承诺
   - 风险：高（当前可能导致UI卡顿）
   - 收益：高（解决根本问题）

2. **JSON序列化原子写入** ⭐⭐⭐
   - 理由：数据安全底线，影响用户信任
   - 风险：中（改进现有逻辑）
   - 收益：高（防止数据丢失）

3. **添加核心模块单元测试** ⭐⭐⭐
   - 理由：当前测试覆盖为0，后续迭代风险高
   - 风险：低（新增文件）
   - 收益：高（建立质量基线）

4. **app.zig拆分重构** ⭐⭐
   - 理由：代码可维护性，为后续功能扩展做准备
   - 风险：中（大规模重构）
   - 收益：中（长期收益）

### P1（重要，1-2周内处理）

1. **搜索/过滤功能** - 用户最容易感知的价值
2. **执行历史记录** - 提升用户体验
3. **脚本分组/文件夹** - 管理大量脚本时必需
4. **环境变量配置** - 脚本执行的必要功能
5. **错误处理系统** - 代码质量基础
6. **日志系统** - 调试和监控基础

### P2（可选，1个月后处理）

1. **缓存新鲜度优化** - 性能微优化
2. **卡片渲染优化** - 性能微优化
3. **FSEvents监听** - 高级性能优化
4. **主题切换** - UX增强
5. **自定义快捷键** - UX增强

- 关键改动：
  - 使用 std.posix.poll() 替代 child.wait()
  - 实现非阻塞IO读取

**Task 1.2: 实现JSON原子写入**

### 9.2 关键坑点提醒

#### 坑点 1：渲染循环内存分配（高风险，影响 Task 3.1-3.3）
- 必须使用栈缓冲区，不能使用 `allocPrintZ()`
- 验证清单：检查所有新组件中是否有动态分配
- 缓解方案：代码审查时重点检查渲染函数中的 `allocator.alloc()` 调用

#### 坑点 2：Tab 状态管理（中风险，影响 Task 3.1-3.3）
- 必须保留 `pending_tab_switch` 机制
- 验证清单：测试 Tab 切换功能是否正常
- 缓解方案：保留原有 Tab 切换逻辑，不要重构

#### 坑点 3：字符串类型混淆（低风险，影响 Task 3.1-3.3）
- 显示文本用 `bufPrint()`，ID 用 `bufPrintZ()`
- 验证清单：编译时检查字符串类型
- 缓解方案：使用 `_id` 后缀命名 sentinel-terminated 字符串

## 十、总结与建议

### 核心结论

两份文档的建议**方向一致但关注点不同**：

- **Codex** 关注**稳定性和可靠性**（执行器、配置、缓存）
- **Claude** 关注**代码质量和可维护性**（拆分、测试、架构）

**两者都重要**，但应该**先稳定后优化**。

### 推荐策略
